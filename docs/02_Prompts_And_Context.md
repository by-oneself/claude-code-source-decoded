# 提示词工程与上下文护盾 (Prompts & Context)

在单纯的开发视角中，大模型的 Prompt 无非是一大长串写死的字符串。但在工业级智能体中，这是系统能否被称得上“智能”的核心壁垒。

Claude Code 将 Prompt 封装得极度结构化并且完全动态，其最重要的实现在于 **Context Collapsing（上下文折叠算法）**。

---

## 1. 结构化拼接 (System Prompt 构建树)

在 `src/utils/systemPrompt.ts` 及各 Tool 文件夹底下的 `prompt.ts` 可以发现，系统提示词并不是一个超级巨大的单体文本，而是以积木形式动态计算的。

1. **基础人设与铁律**：例如“你是 Claude，一个顶级工程师……”位于基础层。
2. **全局上下文注入**：如果当前工作目录下有特定的 Lint 规则、或者 `.claudecode` 引导配置文件，会被提取并拼接为 XML `<project_rules>`。
3. **工具能力的动态挂载**：根据当前权限配置（是否开启 YOLO，是否启用代码修改工具），将允许使用的 Tool 列表及其内部限定（通过 `prompt.ts` 暴露）动态拼接组装。
   * 例如 `BashTool/prompt.ts` 中写着极度严苛的使用警告：“你能够看到文件但不能擅自删除根目录，必须将复杂调用写入单独文件再执行”。

这套树状的组装机制让大模型在极短的字数内得到了清晰严格的行动边界划分。大量的 XML （`<plan>`, `<tool>`, `<file_content>`) 的闭合使用，充分发挥了 Claude 系列大模型对 XML 原生的高保真解析能力。

## 2. 杀手锏：上下文折叠压缩 (Context Collapsing)

位于 `src/utils/messages.ts` 与 `src/services/compact/compact.ts` 中的算法，是这份源码最值钱的机制之一。

### 为什么需要折叠？
当 Agent 被赋予写一个中大型脚本的任务时，它可能会发生长达 30 轮的试错拉扯：
* `Agent`: 跑 `pnpm run test` -> 报错 -> 我看到个错，去看看源码
* `Agent`: 看完源码，发现少了依赖 -> 跑 `npm install xxx` -> 报错
* `Agent`: ...

这个冗长的试错过程一旦填满 10 万或 20 万 Token，除了 API 账单直接起飞外，更可怕的是会导致大模型的“注意力分散”。前面的错乱信息会严重污染后期的推理。

### Claude Code 的做法
在 `compact.ts` 等逻辑中，一旦上下文 Token 逼近预设红线，或是任务进入下一阶段（比如解决了一个大 Bug 后要接续下一个任务前），系统就会进行**隐式折叠**：

1. **提取高亮摘要**：后台调度一次轻度 LLM 询问或启发式代码，把过去这 30 轮的“扯皮对话”总结为类似：`<summary>之前因为无依赖报错，已使用 npm install 安装了 lodash</summary>`。
2. **清洗执行树**：强制修剪掉聊天记录（Message Array）中间那些一整块无意义报错输出（stdout 被截断抛弃）和冗乱思考过程（Thinking Blocks）。
3. **保留决策锚点**：系统依然保留着用户的原始意图消息（最早的 User Request），并保留着系统最新的文件树状态快照。
4. **重建“脑区”**：把这份极限精简后的历史数组重新发给大模型。在大模型看来，它依然觉得自己是一路聊过来的，但没有了中间干扰。

这也就是为什么 Claude Code 可以连续工作几个小时而不崩溃或降级的根本原因。

---

## 3. 防跑飞防御策略

系统在 `messages.ts` 等文件中预置了一系列专用的 `<caveat>` 标签，用于控制系统指令不会被意外篡改（防 Prompt 注入）。例如，用户输入如果含有极强的情感色彩诱导或者指令冲突，内置的特定 Tool 返回结构（甚至是一些 `Reject Message` 和 `Auto_Mode_Denial_Prefix`）能够强制把大模型的乱推导硬掰回工具链逻辑上来。
