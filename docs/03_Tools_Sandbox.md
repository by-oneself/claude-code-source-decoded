# 神级兵器库：工具与沙盒编排 (Tools & Sandbox)

如果你正在使用普通的 LLM 服务或初级的开源 Agent 框架（如 LangChain 的无脑封装），你会发现执行外部工具异常脆弱。`执行 Shell 极易卡死`、`返回内容太大就爆系统`、`一跑测试就陷入死循环`。

进入 `src/tools/` 目录，看看业内天花板的大牛们是如何解决这三个世纪难题的：让大模型真正像黑客一样操控电脑。

---

## 1. 核心底座：`BashTool` 绝不上吊原理

传统的 Bash 执行，是直接跑一个 `child_process.exec()`，大模型说干什么就干什么。这有致命问题：大模型要是跑了个 `npm run dev` 这种会永远挂起的常驻进程，Agent 就当场去世（死等结果永不返回）。

### 进程脱离与常驻 Shell 沙盒
由于需要保留上下文状态，比如允许 `cd src` 接着运行下一个命令，Claude Code 不是跑一次新建一个进程，而是用 PTY / Spawn 建立了一个**常驻后台 Bash Session**。

在 `src/tools/BashTool/` 下可以看到他们的精妙设计：
1. **输入隔离**：当模型下发命令后，系统会将命令拼接一层特殊的包裹标记传递给由于长期运行着的 Bash PTY。
2. **死循环熔断与实时流**：启动一整套监听（Streaming Reader）。当命令执行时长超出某个阈值时，并**不等他彻底死掉**，而是强制返回，并将到目前为止截获的最后几行 stdout/stderr 附加上一条：“由于执行时间过长，此为进程截止目前的截取快照。如果这是一个服务端，请将服务放入后台”。
3. **输出体积防御**：如果一条命令输出了百万行的乱码，系统会有内置的滑动窗口限流器，裁剪冗余，只保留头部或者尾部的报错结论传回。

## 2. 代码篡改手术刀：`FileEditTool`
大多数玩具级 Agent 只会一种姿势：把文件全文发给 LLM，LLM 整体重写一遍再发回来，然后本地覆写。耗时极长且极易因为代码被 Token 拦腰截断而破坏全量部署。

`src/tools/FileEditTool/FileEditTool.tsx` 告诉了你工业级的解法：
* 模型只需声明 `<TargetString>` 和 `<ReplacementString>`。
* 并在 Tool 调用里声明**行号**范围（甚至容忍模糊行号）。
* JS 引擎将在本地做极其苛刻和安全的文本差异比较（Diff/Patch）。
* 对合并内容进行严谨的边界校验后，只替换那 3 行改变了的代码。

这避免了让昂贵且易碎的大模型重新把那几千行原封不动的代码“背诵”一遍。

## 3. 分形子代理：“套娃”的 `AgentTool`
这是让开发者惊呼神迹的一个大巧不工的 Tool。

在处理长周期任务（比如部署一个跨端服务并写单元测试）的时候，大模型常常会被小 Bug 缠住（Token 迷雾）。如果使用 `AgentTool`：

1. **主 Agent 断点**：主 Agent 将任务进行宏观分块后，在终端调用 `AgentTool_Explore`（探路斥候 Agent）。
2. **生成虚拟沙盒**：此时一个全封闭全新上下文的幽灵 Sub-Agent 启动。
3. **独立攻坚**：子 Agent 会尽情在大约几万 token 甚至循环十几轮的环境里研究“这段测试代码为什么会报 500 错”，去翻阅代码乃至翻阅整个项目架构。
4. **结论收敛**：研究完毕后，子 Agent 只向上返回一条消息。这对主 Agent 而言，就是用极短的 1 次 Tool 调用成本，换取了后台可能耗费上万个 Token、数百步循环而最终得出的确切成果。

这种高度封装的、可以带深度隔离树状执行逻辑的工具设计模式，极大地拓宽了复杂业务中 LLM “有效推理上限”。
